# Voice E2E Real-Speech Test Plan

## 1. Goal

Add a realistic speech validation layer to complement the existing deterministic synthetic-fixture suite (`S1-S6`), so we can catch regressions caused by accents, microphone artifacts, noise, and natural speaking patterns.

This plan does not replace the current synthetic suite. It adds a second suite tier with stronger real-world coverage.

## 2. Why We Need a Separate Tier

Current fixtures in `/Users/hao/Projects/teanode-pipeline/test/voicee2e/fixtures` are generated by TTS and are intentionally stable for CI reproducibility. They are good for protocol and turn-control regression checks, but weak for:

1. Accent and dialect variability.
2. Hesitations, fillers, and self-corrections.
3. Background noise and channel distortions.
4. Real pacing that impacts VAD and turn segmentation.

## 3. Scope

In scope:

1. New real-speech fixture library with metadata.
2. New scenario suite file for real-speech runs.
3. New assertions and thresholds tuned for natural speech variance.
4. Two execution modes:
1. `smoke` (PR/non-blocking initially)
2. `full` (nightly/release blocking once stable)

Out of scope:

1. Human labeling platform build-out.
2. Replacing current `S1-S6` synthetic fixtures.

## 4. Real-Speech Fixture Dataset

Create `/Users/hao/Projects/teanode-pipeline/test/voicee2e/fixtures_real/` with versioned metadata.

### 4.1 Recording targets

1. Speakers: at least 12 unique speakers.
2. Accent coverage: at least 6 US/international accent groups.
3. Gender balance: roughly balanced.
4. Speech styles: concise command, medium question, long explanation, interruption phrases.

### 4.2 Audio conditions

Each base utterance should have variants:

1. Clean close-mic.
2. Laptop mic room echo.
3. Low-level background noise (cafe/home).
4. Mobile/compressed channel.
5. Quiet speaker / low volume.

### 4.3 Clip specifications

1. WAV PCM16 mono.
2. Keep native sample rate in source; harness resamples to 16k.
3. Duration buckets:
1. Short: 1-3s.
2. Medium: 4-8s.
3. Long: 9-20s.

### 4.4 Metadata schema

Create `/Users/hao/Projects/teanode-pipeline/test/voicee2e/fixtures_real/metadata.json`:

1. `fixture`
2. `canonical_text`
3. `speaker_id`
4. `accent`
5. `style` (`short|medium|long|interrupt`)
6. `environment` (`clean|room|noise|mobile|quiet`)
7. `snr_bucket` (`high|med|low`)
8. `license` and consent proof reference

## 5. Scenario Design

Add `/Users/hao/Projects/teanode-pipeline/test/voicee2e/scenarios/real_speech_suite.yaml`.

### 5.1 Scenario families

1. `rs1_short_clean`:
1. Equivalent of S1 with human speech.
2. Validates baseline commit and response path.

2. `rs2_medium_accented`:
1. Medium turn from non-default accent.
2. Validates transcript robustness + latency.

3. `rs3_long_natural_pauses`:
1. Long utterance with pauses/fillers.
2. Guards against premature commit.

4. `rs4_multi_turn_context`:
1. Two turns from same speaker.
2. Validates context carry-over under natural variability.

5. `rs5_barge_in_real`:
1. Human interruption over active response.
2. Validates stop latency and next-turn correctness.

6. `rs6_rapid_interrupt_real`:
1. Two fast interruptions.
2. Validates no deadlock/race regressions.

7. `rs7_noise_resilience`:
1. Noisy utterance with same canonical text as clean pair.
2. Measures degradation delta vs clean baseline.

8. `rs8_quiet_voice`:
1. Low-volume speech.
2. Validates VAD sensitivity and transcript presence.

### 5.2 Pairing strategy

For at least 4 intents, include clean/noisy paired fixtures. Gate on relative degradation, not only absolute score.

## 6. Metrics and Gates

Real-speech thresholds should be slightly looser than synthetic thresholds but must be stable over repeated runs.

### 6.1 Functional gates

1. `transcript_count > 0` for all scenarios.
2. `response_count > 0` for all scenarios.
3. Required event ordering remains valid.
4. For interruption scenarios, `barge_in_count > 0`.

### 6.2 Transcript quality gates

1. Per-scenario similarity floor:
1. Clean: `>= 0.50`
2. Noise/mobile/quiet: `>= 0.40`

2. Pair degradation gate:
1. `(clean_similarity - noisy_similarity) <= 0.18`

### 6.3 Latency gates

1. `latency_speech_end_to_transcript_ms`:
1. Clean p50 `<= 5000`
2. Noisy p50 `<= 7000`

2. Barge stop latency (`max_barge_stop_ms`):
1. p95 `<= 650ms`

### 6.4 Stability gates

Run each real-speech suite 3 times:

1. No hard failures in any run.
2. Similarity p50 variance across runs `<= 0.08`.
3. Latency p50 drift across runs `<= 20%`.

## Phase 2: Add gates and CI (non-blocking)

2. Extend report summary with clean/noisy grouped metrics.

Exit criteria:

1. 7 consecutive nightly passes or explained failures.
2. Thresholds tuned from observed distribution.

## Phase 3: Promote to blocking

1. Make `real-smoke` blocking for release branches.
2. Keep full `real` suite nightly + pre-release.

Exit criteria:

1. No unresolved flake scenario.
2. Documented baseline and drift alerts active.

## 8. Ownership and Workflow

1. Dataset owner: voice QA.
2. Harness owner: voice platform team.
3. Review cadence:
1. Weekly threshold review for first month.
2. Monthly dataset expansion and retirement of weak fixtures.

Change policy:

1. Any new real fixture requires metadata entry and license/consent reference.
2. Threshold changes require baseline comparison from at least 3 prior nightly runs.

## 9. Risks and Mitigations

1. Flakiness from natural variance.
1. Mitigation: repeated-run median gates and degradation-based comparisons.

2. Dataset privacy/compliance issues.
1. Mitigation: explicit consent tracking and approved storage location.

3. CI runtime increase.
1. Mitigation: keep `real-smoke` small; full suite nightly.
