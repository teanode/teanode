# TeaNode Generative UI MVP Implementation Brief

Implement a TeaNode-native generative UI MVP in this worktree.

## Context

We do **not** need backward compatibility for older artifact/chart/suggestion text-marker conventions at this stage. Favor the new structured path directly where practical.

TeaNode currently has websocket/RPC conversation events, tool activity, approval/question panels, message rendering, and a React web frontend.

Goal: add first-class structured render payloads, surfaces, and unified interrupts, with a minimal but working vertical slice across backend + frontend.

## Desired MVP scope

### 1. Structured render payloads

Add support for conversation events carrying structured render payloads, at minimum:
- `surface`
- `interrupt`
- optionally `artifact`, `chart`, `suggestions` if easy, but surface/interrupt are the priority

### 2. Surface model

Implement a small schema-driven surface system with support for:
- `surfaceId`
- `schemaVersion`
- `location`: start with `inline` and `right_panel`
- `title`
- `components`

Initial component catalog should be intentionally small:
- `Section`
- `Markdown`
- `KeyValueList`
- `Table`
- `StatusBadge`
- `ButtonRow`
- `Form`
- `CodeBlock`
- `Timeline`

Form fields can be minimal:
- `TextInput`
- `Textarea`
- `Select`
- `Checkbox`

### 3. Unified interrupt model

Implement a generalized interrupt type that can represent at least:
- `question`
- `approval`
- `choice`
- `form`
- `review`

Refactor the frontend so pending questions and approvals are presented through a shared interrupt rendering path, even if some existing components are reused under the hood.

### 4. Surface rendering in frontend

Add a `SurfaceRenderer` and wire it into the conversation UI.

Requirements:
- inline surfaces render near associated conversation content
- right-panel surfaces render in a panel area (it is okay to reuse/adapt current artifact panel plumbing if convenient)
- support basic button actions and form submission UI

### 5. User action round-trip

Add a structured action path from frontend back to backend for surface actions, e.g. `surface_action` / equivalent RPC method.

MVP behavior is acceptable if action handling is simple, for example:
- routes through backend runtime/session layer
- can inject a synthetic user-visible event/message or resume a pending run path
- does not need a fully general patch engine yet

### 6. Minimal backend emission path

Implement at least one simple way for the backend to emit a surface and/or interrupt so the stack is end-to-end testable.

If a broad tool/runtime integration is too much for one pass, it is acceptable to:
- add a dedicated RPC/debug/demo path, or
- hook into an existing conversational/tool path in a narrowly scoped way

But prefer a design that is not purely throwaway.

## Architectural preferences

- TeaNode-native design; do not pull in CopilotKit or external AG-UI/A2UI runtime dependencies.
- Schema-driven only; no arbitrary HTML/JS execution from model output.
- Keep naming explicit and aligned with repository naming rules.
- If adding Go receiver methods, use `self`.
- Prefer clear names like `surfaceActionRequest`, `pendingInterrupt`, etc.
- Errors should be named `err` where possible.
- Avoid single-letter variables.

## Files likely involved

Backend likely touches some of:
- `internal/api`
- `internal/runners`
- `internal/tools`
- `internal/store`
- `internal/web` or websocket/RPC plumbing

Frontend likely touches some of:
- `web/src/types.ts`
- `web/src/hooks/useBackend.ts`
- `web/src/components/*`
- conversation route/components

## Acceptance criteria

- New structured types exist for surfaces and interrupts on both backend and frontend.
- Frontend renders at least one surface end-to-end from backend data.
- Frontend can send a surface action back to backend.
- Unified interrupt path exists and is wired into UI state.
- Build, format, lint, and tests pass, or any unavoidable failure is clearly documented.

## Workflow requirements

1. First inspect the relevant backend/frontend code and write a concise implementation plan to a file in the worktree, e.g. `.claude-progress.md`.
2. Then implement the changes.
3. Run formatters and linters before finishing:
   - backend: `gofmt` and linter/test commands used by repo
   - frontend: formatter/linter/test/build as appropriate
4. If something cannot be completed cleanly, document remaining issues and tradeoffs in `.claude-progress.md`.

## Deliverables

When done, provide:
- summary of files changed
- concise explanation of architecture choices
- commands run and results
- any remaining issues / follow-ups
